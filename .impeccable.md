# Kenji — Design Context

> This file is the short brief for anyone (human or agent) about to touch the UI.
> The full system — tokens, scale, component anatomy, motion specs, voice rules —
> lives in **[DESIGN.md](./DESIGN.md)**. Read that before shipping a component.
> This file tells you the *who*, *why*, and *feel* so the spec doesn't get
> applied in a vacuum.

## Design Context

### Users

Kenji is used by two overlapping groups, in this order of priority:

1. **The operator** — the person actively running a job search. They care about
   one thing: finding the small number of roles that are genuinely worth their
   time, then getting an application out without losing an afternoon. They are
   not here to browse. They sit with the product in short, focused sessions
   (15–45 min), often late in the evening after work. They want to feel that
   progress is being made *for* them while they were away.

2. **Non-technical friends using a hosted instance.** They are not going to
   self-host. They have never heard of Greenhouse or Ashby. They arrive
   curious, need the interface to teach them what it does, and will leave if
   the first screen looks like a database.

The operator's workflow is the source of truth. Everything has to work for
them first. But empty states, copy, and navigation must be clear enough that a
non-technical user can land on the Discover page and immediately understand:
*something is being ranked for me, and I can talk to it.*

**Jobs to be done**, in priority order:

1. Surface the 3–10 roles out of hundreds that are actually worth reading.
2. Let me tell the agent what I want, in my own words, and have it act.
3. Tailor a résumé/cover letter grounded in real stories I've written — never invent.
4. Tell me when to submit, tell me what's pending approval, stay out of my way otherwise.
5. Remember what I applied to, who I heard back from, and what's next.

### Brand Personality

**Composed. Grounded. Literary.**

Kenji should feel like a well-edited magazine feature about your job search, not
a dashboard of it. It is measured. It uses italic emphasis (in sage) the way a
good editor underlines a key word — sparingly, and always to make a point. It
never shouts. It never bounces. It does not reward you with confetti or streaks.

The interface is a *serene, high-end workspace*. Think: an architect's studio at
6pm. Natural light. A single good pen on the desk. Nothing on the wall except
what's being worked on. The opposite of a job board. The opposite of a
productivity app.

**Emotional goals:**

- *Confidence* — the operator never wonders what the agent just did.
- *Calm* — visiting the app should lower, not raise, the heart rate.
- *Trust* — every claim on a résumé traces to a story; the user approves every
  outbound action.
- *A quiet pride* — the craft of the interface itself signals that the work
  inside it is being done with care.

### Aesthetic Direction

**Editorial minimalism in an earth palette.** Warm off-white canvas, deep sage
as the agent voice, muted forest as structure, terracotta reserved exclusively
for human-in-the-loop / approval moments. Never neon. Never purple-to-blue.
Never glassmorphism as decoration.

Typography is the primary design device: **Manrope** for display and headlines
(tight tracking, italic emphasis), **Inter** for all UI text, **JetBrains Mono**
as the "machine voice" for tool calls, IDs, timestamps. Type does the work
that gradients and glow effects do in other AI products.

Full token set, type scale, component anatomy, and motion rules are in
**[DESIGN.md](./DESIGN.md)**. Do not re-derive them.

**Theme:** Light only, by principle (see DESIGN.md §11). Dark mode is
intentionally deferred — inverting surfaces would break the sage↔terracotta
relationship against a warm near-black.

**Anti-references** — Kenji must visibly NOT resemble:

- **LinkedIn / Indeed** — blue-heavy, notification-count dopamine hooks,
  infinite social feed, banner ads, "who viewed your profile" grids, skill
  endorsement spam. Kenji shows a small number of ranked roles, not a
  firehose.
- **Enterprise ATS / HRIS** (Greenhouse admin, Workday, Lever dashboards) —
  dense data tables, gray-on-gray hierarchy, tab bars inside tab bars, UI
  copy written by a compliance committee. Kenji is a tool *for* the
  candidate, not an org chart.
- **Generic AI SaaS** — purple-to-blue gradient text, glowing orbs, "Powered
  by AI" badges, dark mode with cyan accents, sparkles everywhere. Kenji is
  warm, specific, and grown-looking. It does not advertise that it uses an
  LLM; it uses one the way a good tool uses electricity.

### Design Principles

Derived from DESIGN.md §1 plus the answers above. Every PR touching the UI is
checked against these five:

1. **Organic intelligence.** The app is smart, but should feel *grown*, not
   generated. Warm off-whites and earth tones over cold gradients. Every
   "clever" detail should look like something a designer decided, not
   something a model defaulted to.

2. **Less but better.** Extreme clarity and breathing room. If an element
   isn't serving one of the five jobs-to-be-done, it shouldn't be on screen.
   Reduce cognitive load at every step. One primary CTA per view. One italic
   accent per paragraph. One terracotta moment per screen.

3. **Groundedness over polish.** Every surface the user sees — a résumé
   bullet, a cover-letter sentence, a match reason — must trace back to
   something the user actually wrote (a story) or something the agent
   verifiably did (a tool call). No invented claims, no vague "Kenji thinks
   this is a good fit." Show the trace.

4. **Approval-first.** The agent announces, the user approves. Never hidden
   autonomy. An approval-pending state is the only moment terracotta takes
   the foreground — it's a visual promise that the agent will not proceed
   without a human.

5. **Teach on empty.** Because non-technical users will land here,
   empty states must explain what this screen does in a single sentence of
   body copy and offer one CTA. No illustrations, no onboarding carousels —
   a sentence and a button, in the product's voice.

### Accessibility

Target is **WCAG 2.2 AA**, with these explicit additions:

- **Reduced motion**: `prefers-reduced-motion: reduce` kills fade-ins, stagger,
  typing dots, shimmer, and pulse. The interface must still communicate agent
  state (use `aria-live="polite"` on the thread and a static "working…" label
  instead of animated dots). No motion animation is load-bearing for
  comprehension.
- **Color-blind safe**: No state (active/waiting/approved/rejected/pass/fail)
  may be conveyed by color alone. Pair color with an icon, a label, or a
  position. This is already codified in DESIGN.md §11 — enforce it in
  reviews.
- **Focus visible**: Every interactive surface has a ≥2px focus outline in
  `--outline` or `--primary`. No `outline: none` without a replacement.
- **Contrast**: Body text ≥ 4.5:1 against its surface. The warm off-white
  canvas is forgiving; verify any text placed on `--sc-high` or
  `--primary-tint`.
- **Keyboard**: The four-screen shell (Discover, Command, Resume Lab,
  Stories) must be fully operable from a keyboard, including the agent
  composer, the approval bar, and the Before/Diff/After toggle.

### What "good" looks like

A change is on-brand when:

- You can't tell which LLM wrote the copy.
- Someone glancing at a screenshot would ask *"what tool is that?"* — not
  *"which AI product is that?"*.
- The non-technical friend opens Discover and understands, in under 10
  seconds, that the app is ranking jobs for them.
- The operator, mid-flow, never has to guess what the agent just did or is
  about to do.

A change is off-brand when:

- It adds a second primary CTA to a view.
- It uses terracotta for anything other than approval / human-in-the-loop.
- It animates something that wasn't previously in need of motion.
- It introduces a gradient on text, a glass card, or a dark-mode accent.
- It adds a "decorative" icon above a heading.
